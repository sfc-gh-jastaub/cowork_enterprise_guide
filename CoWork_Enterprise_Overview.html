CoWork Enterprise Deployment Guide

The Logic Behind the Notebooks

An overview of why the deployment is sequenced the way it is — and how each notebook sets up the next — for taking a Snowflake CoWork (Snowflake Intelligence) agent from raw tables to a governed, monitored, business-user-ready deployment.

🎯 Mission: a production runbook for standing up an enterprise's first agentic-AI use case on Snowflake — not a feature tour.
How to read this. This is the narrative companion to the nine hands-on notebooks — a walk through the reasoning, not a step checklist. Each notebook below is framed as the question it answers → the logic → the key decisions → what it hands to the next notebook. If you want the runnable steps, open the notebooks themselves; if you want to understand the shape of the whole thing, read straight down.

Three ideas that explain every decision

Almost every choice in these notebooks follows from three principles. Hold these and the ordering stops feeling arbitrary.

1 · The agent runs with caller's rights

An agent has no service account. Every tool call and generated query executes as the invoking user's role — inheriting their grants, masking, and row-access. So there is no separate "AI security layer": getting RBAC right is your CoWork security posture.

2 · Govern → Build → Go Live

You establish meaning and access before the agent exists, so the agent is built into an already-governed environment rather than having governance bolted on afterward. Data → context → access → agent → cost → go-live → improve.

3 · Evaluation is the gate

Nothing is promoted or published on vibes. A scored evaluation (Agent GPA) gates go-live, gates dev→prod promotion, and gates every continuous-improvement change. The threshold is the decision-maker.

+ · Safe on a shared account

A cross-cutting rule: operations that touch only our own objects are runnable; operations that would affect shared state (the singleton CoWork object, account settings, other teams) are shown as reference. More on this at the end.

Walking the logic, notebook by notebook

Each block: the question it answers, the reasoning, the decisions that matter, and how it feeds forward.

00Phase 0 · Foundation

Lab Setup

"What do I build on?"

The logic. In a real engagement, Phase 0 is to confirm the mature estate a customer already runs — RBAC, network policy, MFA, masking. There's no estate to confirm in a demo, so this notebook instead builds a clean, isolated sandbox you can safely tear down: one database, three schemas, one dedicated role, an XS warehouse, and self-contained sample data (a fictional capital-markets firm).

Key decisions

  • Everything lives under one database + role, so the whole demo is grantable and droppable as a unit — isolation on a shared account.
  • Cross-region inference is enabled here so later Cortex features "just work."
Hands off → a governed namespace and the raw tables that the context layer will give meaning to.
01Phase 1 · Meaning

Build the Context Layer

"How does the agent know what the data means?"

The logic. An agent is only as good as the meaning layer beneath it. Raw tables don't tell an LLM that "AUM" is assets under management or how clients join to trades. The semantic view encodes that — logical tables, relationships, facts, dimensions, business metrics, and verified example queries — so the agent generates correct SQL instead of guessing. A second Cortex Search service indexes unstructured research notes for retrieval (RAG), so the agent can answer qualitative questions too.

Key decisions

  • Fuzzy matching comes first. A Cortex Search service over the high-cardinality CLIENT_NAME column is created before the semantic view, because the view's CLIENT_NAME dimension references it (WITH CORTEX SEARCH SERVICE) so a user typing "Acme" resolves to "Acme Capital Partners LLC."
  • Descriptions are the #1 accuracy lever, relationships must be explicit (the agent won't join what you didn't declare), and verified queries teach correct patterns.
  • One semantic view per domain. A single Analyst call works against one view; tightly-related tables stay together, genuinely distinct domains get their own views.
  • Validate before building the agent. Query the view directly first — if the numbers are wrong here, the agent will be wrong too.
Hands off → a validated semantic view + search services that the agent will be wired to in NB03 — but only after access is governed in NB02.
02Phase 2 · Access

Govern AI Access

"Who is allowed to see what — before anything can consume it?"

The logic. Because the agent runs with caller's rights (principle #1), governing the user governs the agent. This notebook builds the consumer role with the four independent layers of access an agent needs at runtime, and enforces PII at the data layer. It comes before the agent exists on purpose: the agent should be born into a governed schema, and masking must already be in place so there's never a window where raw PII could surface.

Key decisions

  • The four-layer model: (1) platform entitlement — CORTEX_USER + USAGE on DB/WH; (2) the specific agent — USAGE ON AGENT, deferred to NB03 since the agent doesn't exist yet; (3) tools/resources — the search services + semantic view; (4) data — SELECT on the tables.
  • Masking is a data-layer control, not a prompt. An AI_REDACT masking policy on the email column redacts for non-privileged roles. Orchestration instructions ("don't show PII") are not a security boundary — they can be talked around; a masking policy cannot.
Hands off → a least-privilege consumer role (missing only the agent grant) and enforced masking. User provisioning is deliberately deferred to go-live (NB05) — a user is inert until the agent exists and is published.
03Phase 3 · Agent

Build & Harden the Agent

"How do I assemble the agent — and what makes it trustworthy?"

The logic. The agent is defined as code from a YAML specification: the model, an orchestration budget, response and routing instructions, and the tools wired to the semantic view + search service. The specification is the artifact you version and promote later. "Hardening" is the set of properties that make it safe to hand to real users.

Key decisions

  • Tool descriptions are the #1 lever for correct routing — the agent picks tools by reading them. Keep the agent narrow.
  • Agents are serverless — the warehouse must be named in the spec, not inherited from the session.
  • The final access layer lands here: now that the agent exists, tier-2 USAGE ON AGENT is granted, completing NB02's four-layer model.
  • Agent identity = caller's rights, no service account, a securable object with an owner, per-user OAuth for external (MCP) actions, and full USER_NAME attribution in the audit trail. You govern the agent by governing users.
  • Account-level guardrails (prompt-injection / jailbreak) and the audit trail are confirmed here.
Hands off → a working, access-complete agent that answers grounded questions — ready to be made cost-safe.
04Phase 4 · Spend

Control the Cost

"How do I give teams AI access without runaway spend?"

The logic. AI credits come from two meters — warehouse compute (the SQL the agent runs) and serverless LLM reasoning — and can be harder to predict than plain compute. The discipline is Observe → Attribute → Control: see what you're spending, tie every credit to a cost centre, then cap it.

Key decisions

  • Attribute before you budget: tag the agent and its warehouse with a cost-centre tag so one budget captures the whole team.
  • Layered controls: a warehouse statement timeout, a per-agent budget with a notification threshold, a per-user quota, and two team budget models (resource-level vs shared-resource).
  • Enforcement stays in monitoring mode on the shared account — the runnable-vs-reference rule (see below) keeps us from capping other teams.
Hands off → an agent whose spend is visible, attributed, and capped — safe to expose to users at go-live.
05Phase 5 · Launch

Evaluate, Go Live & Operate

"Is it good enough to ship — and how do real users reach it?"

The logic. This is where evaluation-as-a-gate (principle #3) becomes concrete, and where the agent finally meets business users. First prove quality; only then publish, brand, and provision users.

Key decisions

  • Evaluate first: golden smoke tests for fast iteration, then a scored Agent GPA run against a curated dataset. Require a threshold (e.g. answer-correctness ≥ 0.85, zero regressions) before publishing.
  • Publishing is curation: the agent is added to the account's singleton CoWork object. Without curation the dropdown shows every agent a user can access; with it, only your production-ready shortlist. Deep links let you test an agent before curating it.
  • Provision SI-only users here, at go-live. A dedicated business user is created and locked to ai.snowflake.com via ALLOWED_INTERFACES — no worksheets, no SQL, no drivers. This lives at launch (not in NB02) because a user can't do anything until the agent is built and published — "build the agent, then let users in."
  • Operate: reuse the golden questions as a scheduled health check.
Hands off → a live, branded, evaluated agent with business users able to reach it — ready to be treated as a versioned, promotable asset.
06Phase 5+ · Promote

Dev → Prod Promotion

"How do I move a validated agent to production safely?"

The logic. You promote the specification, not the running agent — the spec is the git-style source of truth. Capture it, version it with a traceable comment, and deploy the identical spec to the prod target. The evaluation threshold from NB05 becomes the promotion gate: no pass, no publish.

Key decisions

  • The prod agent is created from the same spec (here in a prod-scoped schema; in reality, a prod account).
  • Clean cut-over: the promoted agent is added to the CoWork object and the dev build removed, so users see exactly one production agent.
Hands off → a production agent under change control — and the pattern for every future release.
07Phase 5+ · Operate

Continuous Improvement

"Go-live is the start — how does the agent keep getting better?"

The logic. A self-improving agent is a governed closed loop with a human in it, not autonomous self-modification. Mine real production traces for failures, fold them into the evaluation dataset, score a baseline, improve the agent's instructions (the primary lever — not more tools), then re-evaluate to prove the change helped with no regressions before promoting.

Key decisions

  • Instructions, not tools, are the main improvement lever — and every change is a versioned, evaluated spec edit.
  • The loop is gated by evaluation and reviewed by a human before anything ships; schedule it (e.g. a monthly task) so the agent improves on real usage.
Hands off → back into NB06 — a validated improvement is promoted through the same gate. The system is now living.
08Tear down

Cleanup

"How do I remove the demo without touching anything shared?"

The logic. Deliberately narrow. It removes only the demo footprint — our agents, budgets, quota, masking policy, the user, database, warehouse, and roles — and detaches our agents from the shared CoWork object without dropping the object. Shared, pre-existing state (the singleton object, PUBLIC grants, account guardrails) is never touched.

Result → the account is back to its prior state; the shared CoWork object is preserved for other teams.

The throughlines that tie it together

Read the notebooks and the same few ideas keep resurfacing. These are what a reader should walk away with.

Governance is inherited, not configured separately

RBAC + masking + row-access flow straight through to agent output because the agent uses the caller's rights. That's why NB02 (access) precedes NB03 (build): the access model is a prerequisite, not an afterthought.

Evaluation is the recurring gate

The same Agent GPA evaluation gates three different moments — go-live (NB05), dev→prod promotion (NB06), and every continuous-improvement change (NB07). "Does it clear the threshold?" is the one question that decides whether anything ships.

The specification is the unit of change

The agent is authored, versioned, promoted, and optimized as a YAML spec — never edited live in isolation. That makes it reviewable, diffable, and safe to promote across environments.

Build first, admit users last

Access scaffolding (the role, grants, masking) is foundational and comes early; actual user provisioning waits until the agent is built and published. You don't hand out keys to a building that isn't finished.

The runnable-vs-reference principle

This guide was built and verified on a shared Snowflake account, which forces a discipline that also happens to be exactly right for a real customer's account.

Kind of operationHow it's treatedExamples
Touches only our own objects, or is scoped by user tags Runnable Our database, role, agent, budgets, per-user quota, the dedicated SI-only user, adding/removing our agent from the shared object.
Would affect shared state, enforce broadly, or harden the account Reference Creating/dropping the singleton CoWork object, revoking CORTEX_USER from PUBLIC, changing account guardrails, network policy / SSO / MFA / PrivateLink.

Account-level hardening that a real deployment must do — but that we don't run on a shared account — includes network policies, SSO/IdP, SCIM, MFA, PrivateLink, and revoking CORTEX_USER from PUBLIC. In a customer's own account, the "reference" operations become runnable; the principle simply says: never let a demo change state that belongs to someone else.
